# 金句首页一言句流改版设计文档

**日期：** 2026-03-22  
**项目：** Golden 金句 App  
**目标：** 将首页改为无顶部区的全屏句子卡阅读流，统一使用一言接口驱动内容，并弱化操作区视觉层级。

---

## 1. 背景与现状

当前首页主实现位于 `src/pages/home/HomePage.tsx`，已经具备纵向句流、当前卡片高亮、收藏/心动/感悟/样式/导出等核心交互，但仍有以下问题：

1. 首页仍依赖 `getQuotes({ page: 1, pageSize: 20 })` 获取首屏数据，内容来源与一言接口割裂。
2. 首页顶部仍有 `HomeHero` 区块，会分散首屏注意力，不符合“直接进入句子阅读”的目标。
3. 底部操作栏位于页面独立区域，按钮存在文字和明显背景，视觉存在感偏强。
4. “下一句”依赖显式按钮触发，破坏纵向连续阅读节奏。
5. loading、错误提示和登录提醒目前更偏页面式展示，不够适配全屏句子卡的沉浸阅读结构。

用户希望首页满足以下方向：

1. 进入首页即看到句子卡，不要顶部标题区。
2. 句子卡可上下滑动，形成连续句流。
3. 首页统一由一言接口驱动。
4. 删除“下一句”按钮，改成自动补句。
5. 保留现有 5 个动作，但做成更小、更轻、只有图标的弱操作区。

---

## 2. 设计目标

本次改版聚焦以下结果：

1. 首页首屏直接进入全屏句子卡阅读，不显示首页顶部区。
2. 首页数据源统一为一言接口，构建可持续下滑的句流体验。
3. 删除“下一句”按钮，改为靠滚动位置自动预取下一句。
4. 保留 `心动 / 收藏 / 感悟 / 样式 / 导出` 五个动作，但显著降低视觉权重。
5. 保持现有登录拦截、感悟、样式编辑、导出等能力可用。
6. 控制改动范围，优先复用已有 `fetchHitokoto` 服务链路，避免扩大 diff。

非目标：

1. 不在本次改版中合并分类页能力到首页。
2. 不在本次改版中新增首页筛选器或推荐算法。
3. 不在本次改版中大改全局导航和其他一级页面结构。
4. 不在本次改版中扩展一言接口分类参数的首页配置 UI。

---

## 3. 方案选择

### 3.1 备选方案

**方案 A：保留首页首屏列表接口，只把末尾补句切到一言**

- 首屏继续调用 `getQuotes`
- 末尾追加时调用 `fetchHitokoto`
- 其余结构维持不变

问题：

1. 首页内容源会继续混用本地句库与一言接口。
2. 首页“流”的语义不纯，用户感知会割裂。
3. 后续还要再做一次切换，重复改动。

**方案 B：首页完全切到一言驱动，保持现有纵向句流骨架**

- 首次进入首页即调用 `fetchHitokoto`
- 继续使用当前的垂直滚动和当前索引逻辑
- 末尾接近时自动预取下一句
- 重写顶部区和操作栏视觉层级

优点：

1. 与用户目标完全一致。
2. 可复用现有 `fetchHitokoto` 前后端链路。
3. 改动集中在首页、操作栏和测试，边界清晰。

**方案 C：改单卡分页式阅读**

- 一次只渲染一张句子卡
- 通过翻页或滚轮控制切句

问题：

1. 虽然更纯粹，但会损失当前“句流”已有实现基础。
2. 行为更接近轮播，不如连续滚动自然。

### 3.2 最终选择

采用 **方案 B：首页完全切到一言驱动，保持现有纵向句流骨架**。

原因：

1. 能最小代价达成“全屏句子卡 + 自动补句 + 弱操作区”的目标。
2. 首页现有滚动切换逻辑可延续，重写成本可控。
3. 数据来源统一后，产品语义清楚，测试也更直接。

---

## 4. 页面与交互设计

### 4.1 页面结构

首页保留单页阅读结构，但移除顶部区，直接由句流容器占据主体：

1. 去掉 `HomeHero` 及其标题、副标题、页内错误/登录提示承载位。
2. 首页主体改为一个接近全屏高度的句流容器。
3. 每张句子卡接近一屏高度，保持 `snap-y` 吸附式纵向阅读。
4. 当前卡高亮，前后卡保持轻微弱化。
5. 操作区悬浮在当前句子卡底部中央，不再占用独立页面区域。

### 4.2 句子卡

`src/components/quote/QuoteCard.tsx` 需要围绕全屏阅读做以下调整：

1. 提高卡片与视口的匹配度，让单张卡的阅读占比更高。
2. 卡片正文区域保留充足留白，使句子成为绝对主视觉。
3. 卡片底部增加安全内边距，避免悬浮操作栏遮挡最后两行正文或作者区。
4. 保留当前可导出的样式容器与 `active` 卡片 ref 机制。

### 4.3 悬浮操作栏

`src/components/quote/QuoteActions.tsx` 改为轻量悬浮工具栏：

1. 删除“下一句”动作及其相关 props。
2. 保留 `心动 / 收藏 / 感悟 / 样式 / 导出` 五个动作。
3. 每个按钮只显示图标，不显示中文名称。
4. 取消每个按钮独立的大面积背景，改成统一胶囊容器内的小图标按钮。
5. 激活态仅通过颜色、透明度、轻微描边或填充变化表达，不喧宾夺主。
6. 仅依赖 `aria-label` 保证无障碍与测试定位。

### 4.4 登录提示与错误反馈

为保持全屏阅读结构，提示不再占用顶部区：

1. 登录受限动作触发后，在首页内部显示轻量就地提示。
2. 错误提示不打断当前句流阅读，只作为弱提示显示。
3. 首次加载使用句子卡形态的占位，而不是通用页面级 loading 模式。

## 5. 数据流与状态设计

### 5.1 数据源统一

首页统一使用 `fetchHitokoto` 获取句子，不再调用 `getQuotes` 作为首页首屏源。

前端服务层优先复用现有接口：

- `src/services/api/quotes.ts`
- `api/quotes/fetch-hitokoto.ts`

当前后端代理已通过 `https://v1.hitokoto.cn/` 拉取一言数据，并将返回映射为项目内部 `Quote` 结构，因此本次无需发明新接口。

### 5.2 首页状态

首页建议保留并调整以下状态：

1. `quotes`：当前已加载句子流。
2. `currentIndex`：当前阅读中的卡片索引。
3. `loading`：首次加载状态。
4. `loadingNext`：自动补句状态。
5. `error`：轻量错误提示内容。
6. `loginPrompt`：登录拦截提示。

### 5.3 自动补句策略

删除“下一句”按钮后，首页通过滚动位置触发自动补句：

1. 首次进入首页时请求 1 句，作为第一张卡片。
2. 当当前索引进入倒数第 2 张附近时，如果当前没有补句请求，则自动请求下一句。
3. 请求成功后，将新句子追加到 `quotes` 末尾。
4. 请求失败时，不清空现有数据，只记录轻量错误提示。

### 5.4 轻量去重

由于一言接口在短时间内可能出现同句返回，首页需要在前端做最小去重：

1. 以 `quote.id` 为主键判断是否已存在于当前句流。
2. 若新句已存在，则继续补拉一次。
3. 为避免死循环，设置有限重试上限。

该策略只负责“当前阅读流内部不重复”，不尝试构建全局句库缓存。

## 6. 组件职责调整

### 6.1 `src/pages/home/HomePage.tsx`

职责将收敛为：

1. 首页首句加载。
2. 纵向滚动索引计算。
3. 自动补句触发与去重。
4. 当前句相关动作分发。
5. 感悟抽屉、样式抽屉、导出能力接入。

需要移除或重构的点：

1. 移除首页顶部 `HomeHero` 展示。
2. 移除显式“下一句”按钮事件处理及文案。
3. 将 `titleCopy` 这类顶部文案状态从页面结构中移除。

### 6.2 `src/components/quote/QuoteActions.tsx`

职责调整为：

1. 作为当前卡片的悬浮工具栏渲染器。
2. 只展示图标按钮与激活态。
3. 不再承载“下一句”动作。

### 6.3 `src/components/quote/QuoteCard.tsx`

职责调整为：

1. 适配整屏卡片尺寸与底部安全区。
2. 继续提供可导出节点 ref。
3. 保证卡片在激活态与非激活态之间的视觉差异可感知但不过重。

## 7. 测试策略

本次改版需要优先更新首页测试 `src/pages/home/HomePage.test.tsx`，建议覆盖以下行为：

1. 首次进入首页时调用 `fetchHitokoto`，而不是 `getQuotes`。
2. 首页不再渲染“获取下一句”按钮。
3. 当滚动到末尾附近时，自动触发补句请求。
4. 自动补句成功后，句流中出现新句子。
5. 自动补句失败时，已有句子仍保留，页面显示轻量错误提示。
6. 收藏、心动、感悟、样式、导出 5 个动作仍然可触发。
7. 操作按钮依旧能通过 `aria-label` 查询，保证测试和可访问性稳定。

如有需要，可补充对 `QuoteActions` 的纯组件测试，但优先保证首页行为测试可读且稳定。

---

## 8. 风险与约束

1. 一言接口存在短时间缓存与重复返回风险，因此首页必须做最小去重。
2. 自动补句若触发条件过早，可能造成无意义请求；过晚则会让用户滑到尽头时出现空档，需要在实现中调好阈值。
3. 去掉顶部区后，登录提示、错误提示和 loading 占位都必须重新安放，否则容易破坏全屏结构。
4. 操作栏弱化后，图标的点击热区仍需足够，避免移动端误触。
5. 首页改成纯一言流后，原首页与分类页的数据语义会更明确，需避免把分类页逻辑重新带回首页。

---

## 9. 实施边界

本次实现预计主要涉及以下文件：

1. `src/pages/home/HomePage.tsx`
2. `src/pages/home/HomePage.test.tsx`
3. `src/components/quote/QuoteActions.tsx`
4. `src/components/quote/QuoteCard.tsx`

如需支持更稳的补句去重，可能小幅调整：

5. `src/services/api/quotes.ts`
6. `api/quotes/fetch-hitokoto.ts`

原则：

1. 优先做小 diff。
2. 不扩散到首页以外页面。
3. 不引入新的全局状态或新路由。

---

## 10. 结论

首页改版将采用“无顶部区的全屏一言句流”方案：

1. 首页首屏直接进入句子卡阅读。
2. 数据统一由一言接口驱动。
3. 通过自动补句替代“下一句”按钮。
4. 通过卡内悬浮图标栏保留动作能力但弱化存在感。

这条路径能在不扩大改动范围的前提下，把首页从“带工具栏的句子列表”推进为“以句子卡为主角的沉浸式阅读入口”。
